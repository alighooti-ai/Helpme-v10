## ۱. یافتهٔ اصلی — مشکل «توقف بی‌پایان» (Infinite Stop Lock)

### نتیجهٔ کلی داده‌ها (v14.10 / GBPNZD / 6 ماه):

| حالت Rule | BUY % | SELL % |
|---|---|---|
| Run | 51.6% | 48.2% |
| Stop | **35.8%** | **34.7%** |
| Close | 12.6% | 17.1% |

عملاً نیمی از وقت در Stop قفل هستیم — اما مشکل **نه از منطق Close است، نه از Crisis سخت‌گیر**. مشکل از جای دیگری می‌آید.

---

### کشف ریشه: مکانیزم قفل در Stop

کد برای خروج از Stop نیاز دارد به **۲۴ بار متوالی H1 بدون «ساعت کثیف»**:

```
hourDirty = (crisis == Red)  OR  (spike >= Warning)  OR  HA_active
```

در دوره‌های Stop، توزیع داده‌ها این است:

| وضعیت در ساعت‌های Stop | BUY | SELL |
|---|---|---|
| Crisis = Green (✓ پاک) | 68.7% | 65.6% |
| Spike = Normal (✓ پاک) | 81.7% | 78.4% |
| **هر دو پاک (کاندیدای CleanH)** | **80.7%** | **77.5%** |
| **ساعت کثیف (از Spike)** | **18.3%** | **21.6%** |
| ساعت کثیف (از Crisis Red) | 1.5% | 2.1% |

**نتیجهٔ ریاضی**: با dirty rate = 19.3% (BUY)، احتمال اینکه ۲۴ ساعت متوالی پاک داشته باشیم:

```
P(24 متوالی پاک) = (1 - 0.193)^24 = 0.806^24 ≈ 0.6%
```

**زمان انتظار مورد انتظار تا Resume:**

```
E[بارها تا خروج از Stop] ≈ 888 ساعت = 37 روز  ← BUY
E[بارها تا خروج از Stop] ≈ 1988 ساعت = 83 روز  ← SELL
```

**به همین خاطر Stop دائمی می‌شود** — نه به خاطر اینکه بازار خطرناک است، بلکه چون Spike Warning با آستانه 1.10 هر ۵ بار یک بار فعال می‌شود و CleanH را ریست می‌کند.

## ۷. بررسی باگ‌ها

### ۷.۱ BUG-HA1 (v14.10 Fix) — وضعیت: ✅ رفع شده
`haSell = ADX >= 35 AND flowSell > +7.0` (غلط) → `flowSell < -7.0` (درست). در v14.10 اصلاح شد. HighAlert برای Sell حالا درست عمل می‌کند.

### ۷.۲ SpikeScore No-Position Display Bug (v14.09 Fix) — وضعیت: ✅ رفع شده
نمایش Telegram در حالت no-position اصلاح شد. Score دو جهت متفاوت نشان می‌دهد.

### ۷.۳ BUG-FLOW-HARDCODE — وضعیت: ⚠️ در v14.11 رفع می‌شود
در چند نقطه از `TL_Update` و `LogHourlySnapshot`، آستانه‌های Flow به صورت `< -4.0` و `< 2.0` هاردکد هستند:
```mql5
bool flowRedBuy = (fsBuy < -4.0);   // هاردکد — باید از Flow_GetThresholds() بخواند
bool flowYellowBuy = (spikeScoreBuy >= 2.0 && fsBuy < 2.0 && fsBuy >= -4.0);  // هاردکد
```
این باگ در v14.10 مشکل ایجاد نمی‌کرد چون آستانه GBPNZD هم -4.0 بود. اما با تغییر به -5.0، اگر هاردکدها اصلاح نشوند، Flow رفتار ناهماهنگ خواهد داشت.

**رفع**: تابع جدید `Flow_GetThresholds(sym, greenTh, redTh)` اضافه می‌شود و تمام نقاط هاردکد به این تابع ارجاع می‌دهند.

- Rule transitions برای Sell: ✅ بعد از BUG-HA1 درست است

## ۸. تغییرات ساختاری v14.11

### ۸.۱ تابع جدید: `Flow_GetThresholds()`
```mql5
void Flow_GetThresholds(string sym, double &greenTh, double &redTh);
```
- در `LogHourlySnapshot()` جایگزین هاردکد می‌شود
- در `TL_Update()` جایگزین `-4.0` و `2.0` می‌شود

### ۸.۲ Inputs جدید — گروه کالیبراسیون
تمام اعداد بالا به صورت `input double` قابل تنظیم از پنل MT5 هستند.  
کامپایل مجدد لازم نیست — در بکتست کافی است عدد را در Inputs تغییر دهید.

### ۸.۳ Placeholder برای AUDCAD و EURGBP
توابع `Spike_GetThresholds`, `Rule_SpikeWeightsForSymbol`, `Crisis_GetThresholds`, `Flow_GetThresholds` برای دو ارز دیگر placeholder دارند (با اعداد v14.10) تا بعد از کالیبراسیون مستقل پر شوند.
